# Project Instructions
## 一、项目架构原则
### （一）分层架构
```
Page → Composable → Service → API
                  │
                  ├── Components（UI）
                  ├── Store（共享状态）
                  ├── Utils
                  ├── Config
                  └── Constants
```

1. Page（页面）只负责页面展示和用户交互，不允许直接请求 API。
2. Composable 负责页面业务逻辑，可组合多个 Service。
3. Service 负责业务编排，可调用多个 API，不负责页面状态。
4. API 只负责 HTTP 请求，不处理业务逻辑。
5. Store（Pinia）仅负责共享状态管理，不处理复杂业务逻辑。

---

### （二）调用规范

1. Page 只能调用 Composable。
2. Composable 只能调用 Service。
3. Service 只能调用 API。
4. API 不允许调用 Page、Composable、Store。
5. 页面之间禁止相互调用。
6. 禁止跨层引用。
---

## 二、项目开发规范
### （一）目录规范
1. 公共业务逻辑统一放入 `composables/`。
2. 公共 UI 统一放入 `components/`。
3. 所有 HTTP 请求统一放入 `api/`。
4. 所有项目配置统一放入 `config/`。
5. 所有业务常量统一放入 `constants/`。
6. 所有共享状态统一放入 `store/`。
7. 模块遵循单一职责（Single Responsibility）。
8. 不允许出现重复代码（DRY）。
禁止因为"当前未引用"而删除产品规划中的预留模块、目录和公共资源。
只有确认属于废弃设计，且未来不会继续使用时，才允许删除。
目录结构属于架构的一部分，应保持长期稳定。
允许文件内容迭代，不允许频繁新增、删除、重命名目录。

---

### （二）编码规范

1. 使用 ESLint + Prettier 统一代码风格。
2. Vue 组件统一使用 `<script setup>`。
3. 样式统一使用 SCSS，颜色、字体、间距等变量统一管理。
4. 命名规范统一如下：

| 类型 | 命名规范 |
|------|---------|
| Page | `index.vue` / `xxx.vue` |
| Component | `PascalCase.vue` |
| Composable | `useXxx.js` |
| Service | `XxxService.js` |
| API | `xxx.js` |
| Store | `xxx.js` |

5. import 顺序统一：

```
Vue
→ 第三方库
→ Components
→ Composables
→ Services
→ API
→ Store
→ Utils
→ Styles
```

6. 所有异步请求统一使用 `async/await`，禁止使用 `then()`。
7. 所有新增页面必须支持 Loading、Empty、Error 三种状态。
8. 注释遵循"说明为什么"，避免解释代码本身。

---

### （三）AI Coding 规范

1. 所有代码必须保证可以直接运行。
2. 不允许生成 TODO、FIXME、占位代码或伪代码。
3. 不允许引用不存在的组件、API、工具函数或资源文件。
4. 不允许遗漏 import、export。
5. 每完成一个模块必须保证可以正常编译。
6. 每新增页面必须同步更新 `pages.json`。
7. 每新增 API 必须同步更新 `spec/tech/openapi.yaml`。
8. 每新增功能必须同步补充对应测试用例。
9. 不允许修改接口返回结构适配业务，接口适配统一在 Service 层完成。
10. 每个开发阶段结束后必须执行 Lint、Build，确保项目可以正常运行。

### （四） UI 样式 要严格使用`spec\ui'

---

## 三、组件与代码规范

1. 优先复用已有组件、Composable、Service、API 和工具，不允许重复实现相同功能。
2. 相同职责全项目只能存在一个实现，禁止创建 `TabBar2`、`AppTabBar`、`NewTabBar`、`V2` 等重复组件或重复模块。
3. 修改功能时必须优先修改现有代码，不允许为了改样式、改布局、改交互而新增组件、文件或抽象层；只有职责发生本质变化时才允许新增。
4. 优先采用最简单、最直接、最易维护的实现方案，避免过度设计、过度封装和无意义的抽象。
5. 在保证可读性、可维护性和可扩展性的前提下，尽可能减少代码量，以最少代码实现完整功能。
6. 优先采用官方推荐方案和社区最佳实践，不使用复杂技巧、黑科技或不必要的设计模式。
7. 废弃的组件、文件、逻辑必须立即删除，并同步清理所有引用、import、路由和无用代码，不允许保留历史版本。
8. 每完成一个功能后，必须检查并清理重复组件、重复逻辑、无引用文件和无效代码，保持项目结构整洁。

**最终原则：遵循 KISS（Keep It Simple, Stupid）原则，以最少的代码、最少的文件、最少的组件实现完整功能。同一种功能全项目只能存在一个实现，保持项目简洁、统一、易维护。**

## 五、重要约束

1. 数字人功能默认关闭，通过 `dhEnabled` 开关控制。
2. 数字人组件采用 `v-if` 懒加载，不影响首屏体积。
3. 开发阶段统一使用 `api/mock/` 数据，不依赖后端。
4. `spec/tech/openapi.yaml` 必须与实际接口保持同步。
5. `spec/` 目录下所有规范文档必须存在且内容不能为空。
6. 用户权限统一采用三级权限模型：
   - `guest`（游客）
   - `user`（注册用户）
   - `member`（会员用户）
7. 页面访问、功能权限、会员能力统一通过 `permission` 模块管理，禁止在页面中硬编码权限判断。
8. 新增功能必须遵循现有项目架构和目录规范，不允许随意新增新的架构层或目录。
9. 修改已有功能时，禁止新增同职责组件或模块，必须优先修改原有实现。
10. 所有代码必须保证能够在微信小程序、H5 环境正常运行，不允许引入平台不兼容实现。
