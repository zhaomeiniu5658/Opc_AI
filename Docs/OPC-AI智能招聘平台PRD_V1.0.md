## 00. 文档说明

### 0.1 版本目标

PRD V1.0 面向 OPC MVP 阶段，主要依据《MVP功能清单.md》和《智能招聘OPC-AI 平台需求完善-0701.md》整理，目标是输出一份可进入研发排期与测试验收的产品需求文档。

本 PRD 按产品研发生命周期组织：业务背景 -> 需求分析 -> 产品设计 -> 技术设计 -> 开发实施 -> 测试验收。后续评审时需按章节逐章确认，任何需求范围、业务规则、权限边界、数据口径的变更必须进入版本变更记录。

## 01. 项目背景

### 1.1 行业背景

CRO、医药临床研究、生物科技行业具备岗位专业度高、候选人圈层封闭、人才流动频繁、隐性人脉影响大的特点。通用招聘平台对 CRA、PM、DM、RA、统计、医学写作、MSL 等岗位理解不足，招聘流程仍依赖人工筛选、经验判断和猎头触达，效率低且数据难沉淀。

### 1.2 业务痛点

| 对象 | 痛点 | 影响 |
| --- | --- | --- |
| 求职者 | 不知道自己适合哪些岗位，简历与经历无法被平台准确理解 | 推荐粗糙、投递效率低、职业规划缺少反馈 |
| 企业 HR | JD 与真实招聘偏好脱节，简历初筛耗时，候选人质量不稳定 | 招聘周期长、沟通成本高、候选人体验差 |
| 猎头/平台运营 | 候选人关系、企业需求、投递反馈分散在人工经验中 | 数据资产难复用，无法形成规模化服务 |
| 管理层 | 缺少统一运营看板、过程审计和 Agent 执行效果评估 | 难以衡量业务转化、风险与交付质量 |

### 1.3 产品机会

OPC 以求职者小程序为 C 端入口，以企业招聘端和平台管理端为 B 端交付工具，通过 AI 职业画像、岗位画像、人才图谱、智能匹配和多 Agent 工作台构建垂直招聘平台。平台不只做职位展示，而是持续理解人才、岗位和关系网络。

## 02. 建设目标

### 2.1 总体目标

建设一套可运行、可演示、可试运行的 OPC 智能招聘平台 MVP，跑通“求职者建档 -> AI 访谈 -> 简历解析 -> 职业画像 -> 岗位画像 -> 人岗匹配 -> 推荐报告 -> 面试反馈 -> 数据沉淀”的核心闭环。

### 2.2 分阶段目标

| 阶段 | 目标 | 衡量指标 |
| --- | --- | --- |
| V1.0 MVP | 跑通三端和 AI 核心闭环 | 主流程可用率 100%，核心数据可保存和查询 |
| V1.1 增强 | 强化图谱、内容审核、企业认证与推荐报告 | 推荐结果可解释，企业端可完成招聘闭环 |
| V1.5 扩展 | 引入更深 Agent 编排、内容运营、用户运营和数据看板 | Agent 任务可追踪，运营数据可复盘 |
| V2.0 平台化 | 支持企业多租户、商业化套餐、ATS 对接 | 企业客户可规模化使用 |

### 2.3 成功指标

| 指标类型 | 指标 |
| --- | --- |
| 用户侧 | 注册登录成功率 >= 99%，简历上传成功率 >= 95%，画像生成成功率 >= 90% |
| 企业侧 | 岗位发布成功率 >= 99%，JD 解析成功率 >= 90%，候选人匹配结果可解释率 >= 95% |
| AI 侧 | Agent 任务状态可追踪率 100%，结构化输出解析成功率 >= 90% |
| 性能侧 | 推荐响应 P95 <= 2s，图谱查询 P95 <= 3s，简历解析异步任务 95% 在 30s 内完成 |
| 运维侧 | MVP 试运行可用性 >= 99.5%，关键接口具备错误日志 |

## 06. 业务流程

### 6.1 求职者建档与推荐流程

正常流程：用户注册/登录 -> 首页 -> AI 职业顾问访谈 -> 上传简历 -> 简历解析 -> 职业画像生成 -> 人才图谱构建 -> 岗位推荐 -> 查看匹配解释 -> 投递/收藏 -> 模拟面试 -> 生成成长计划 -> 保存历史

异常流程：用户注册失败 -> 提示手机号/邮箱/验证码错误 -> 允许重试；简历解析失败 -> 保存原文件 -> 提供手动补充入口 -> 标记解析失败原因；岗位推荐为空 -> 展示空状态 -> 引导完善画像或扩大城市/岗位偏好

超时处理：简历解析超过 30s 未完成 -> 前端提示“解析中，可稍后查看” -> 后端任务继续执行；AI 对话超过 60s 未响应 -> 返回重试提示并记录 Agent 超时日志

权限校验：仅登录求职者可上传简历、查看个人画像和投递记录；企业不可查看完整简历和联系方式，除非求职者授权。

数据流向：用户输入 -> API 鉴权 -> 文件/表单数据入库 -> AI 解析与画像 -> 匹配服务 -> 推荐结果入库 -> 前端展示 -> 用户行为回流。

### 6.2 企业招聘流程

正常流程：企业注册/登录 -> 提交企业认证 -> 平台审核通过 -> 发布岗位/上传 JD -> JD 解析 -> AI 岗位访谈 -> 岗位画像生成 -> 智能匹配候选人 -> 查看候选人详情 -> 查看推荐报告 -> 预约面试/反馈 -> 招聘结果沉淀

异常流程：企业认证驳回 -> 展示驳回原因 -> 允许补充材料重新提交；JD 解析失败 -> 允许手动录入岗位字段；候选人无授权 -> 展示匿名画像并引导发起授权申请

超时处理：JD 解析超过 30s -> 展示解析任务状态；匹配任务超过 10s -> 返回任务 ID 并异步刷新结果。

权限校验：未认证企业不可发布岗位或查看候选人；企业只能访问本企业岗位和已授权候选人数据。

数据流向：企业资料 -> 审核记录 -> 岗位/JD -> 岗位画像 -> 匹配记录 -> 推荐报告 -> 面试反馈 -> 画像和匹配模型优化。

### 6.3 平台管理流程

正常流程：管理员登录 -> 数据概览 -> 用户/企业/岗位管理 -> 内容审核 -> 图谱审核 -> 匹配监控 -> 系统配置 -> 操作日志沉淀

异常流程：管理员权限不足 -> 返回 403 并记录访问失败；审核对象不存在 -> 返回 404 并提示刷新列表；系统配置保存失败 -> 回滚前端状态并记录错误。

超时处理：大数据量查询超过 5s -> 使用分页、筛选和导出异步任务；图谱查询超过 3s -> 返回部分结果并提示缩小筛选范围。

权限校验：仅平台管理员和授权运营可进入管理端；敏感配置仅超级管理员可修改。

数据流向：管理操作 -> 权限校验 -> 业务表更新 -> 审计日志 -> 相关端通知或状态刷新。

### 6.4 Agent 执行流程

正常流程：用户/运营发起任务 -> 指挥官 Agent 识别意图 -> Workflow 选择节点 -> 专业 Agent 调用 LLM/RAG/MCP/数据库 -> 结构化输出 -> 结果审核/入库 -> 前端展示

异常流程：意图无法识别 -> 返回澄清问题；模型调用失败 -> 进入降级模型或重试；结构化输出解析失败 -> 再次请求模型按 JSON 输出；工具调用失败 -> 标记任务失败并暴露失败原因。

超时处理：LLM 单次调用超过 60s -> 中断并记录；Workflow 总耗时超过 120s -> 返回异步任务状态。

权限校验：Agent 只能访问完成任务所需资源；合规/审计类 Agent 只读，不直接修改业务数据。

数据流向：任务输入 -> Agent 任务表 -> 模型/工具调用日志 -> 结构化结果 -> 业务数据表/报告表 -> 展示与审计。

## 08. 功能模块设计

### 8.1 M-01 账号与权限模块

| 项目 | 内容 |
| --- | --- |
| 功能简介 | 提供求职者、企业用户、平台管理员的注册、登录、身份识别、角色权限和会话管理。 |
| 建设目标 | 保证三端用户可安全进入系统，并按角色访问对应业务能力。 |
| 业务规则 | 手机号/邮箱/微信登录至少支持一种；用户角色包括 candidate、enterprise_user、enterprise_admin、platform_operator、platform_admin、super_admin；未登录用户只能浏览公开内容；账号封禁后禁止登录。 |
| 页面入口 | 小程序登录页/注册页，企业端登录页/注册页，管理端登录页。 |
| 页面结构 | 登录表单、验证码/密码输入、注册入口、协议勾选、错误提示。 |
| 页面交互 | 输入校验、登录提交、注册提交、退出登录、Token 过期自动跳转登录。 |
| 功能说明 | 登录成功后写入会话 Token；后端根据 Token 解析用户 ID 和角色；前端根据角色控制菜单和按钮。 |
| 状态流转 | 未注册 -> 已注册 -> 已登录 -> 已退出；正常 -> 封禁 -> 解封；企业用户未认证 -> 认证中 -> 已认证 -> 认证驳回。 |
| 数据字段 | 用户ID、邮箱、手机号、密码凭证、姓名、角色、所属企业、账号状态、创建时间、最近登录时间。 |
| 权限控制 | RBAC 控制页面、接口和按钮；管理端接口必须校验 platform_admin 及以上权限。 |
| 异常处理 | 密码错误、验证码错误、账号不存在、账号封禁、Token 过期均返回标准错误码和可读提示。 |
| 数据存储 | 用户账号、企业资料、登录会话、Token 黑名单。 |
| API接口 | POST /api/auth/register、POST /api/auth/login、POST /api/auth/logout、GET /api/auth/me。 |
| 埋点需求 | login_view、login_success、login_failed、register_success、logout_click。 |
| 日志记录 | 登录成功/失败、注册、退出、封禁/解封、权限拒绝。 |
| 验收标准 | 不同角色登录后只能访问授权页面；Token 过期后不可继续调用受保护接口；错误提示明确。 |

| 项目 | 内容 |
| --- | --- |

## 13. API 接口需求

### 13.1 接口规范

接口按 OpenAPI/Apifox 规范设计，统一使用 JSON，受保护接口使用 Bearer Token。所有响应包含 code、message、data、trace_id。

成功响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "trace_id": "req_20260709_000001"
}
```

失败响应：

```json
{
  "code": "AUTH_REQUIRED",
  "message": "请先登录",
  "data": null,
  "trace_id": "req_20260709_000002"
}
```

### 13.2 通用错误码

| 错误码 | HTTP | 说明 |
| --- | --- | --- |
| AUTH_REQUIRED | 401 | 未登录 |
| PERMISSION_DENIED | 403 | 权限不足 |
| VALIDATION_ERROR | 400 | 参数错误 |
| NOT_FOUND | 404 | 资源不存在 |
| CONFLICT | 409 | 状态冲突或重复操作 |
| AI_TASK_FAILED | 500 | AI 任务失败 |
| FILE_UNSUPPORTED | 400 | 文件类型不支持 |
| RATE_LIMITED | 429 | 请求过于频繁 |

## 14. 权限设计

### 14.1 权限模型

平台采用 RBAC + 数据权限 + Agent 权限 + 用户授权四层控制。

| 权限层 | 说明 |
| --- | --- |
| RBAC | 控制用户可访问的页面、菜单、接口和按钮 |
| 数据权限 | 控制用户只能访问自己或所属企业的数据 |
| Agent 权限 | 控制 Agent 只能调用授权工具和读取必要数据 |
| 用户授权 | 控制企业是否可查看求职者完整简历、联系方式和关系网络 |

## 18. 风控设计

### 18.1 风险类型

| 风险 | 规则 |
| --- | --- |
| 隐私泄露 | 企业查看简历和联系方式必须校验授权 |
| 虚假企业 | 企业认证材料需人工审核，异常企业可暂停 |
| 虚假岗位 | 岗位描述、薪资、企业主体需审核或抽检 |
| AI 幻觉 | AI 输出需基于简历、岗位、知识库或明确标注推断 |
| Prompt 注入 | 用户输入不得直接覆盖系统 Prompt 或越权调工具 |
| 外部情报误判 | 公开采集数据默认待审核，低置信度不可进入业务决策 |
| 恶意爬取 | 列表接口分页、限流、登录态、敏感字段脱敏 |
| 文件安全 | 限制文件类型、大小和解析沙箱 |

### 18.2 风控动作

正常放行 -> 记录日志；中风险 -> 展示提醒并进入待审核；高风险 -> 拦截操作、记录审计、通知管理员。

### 18.3 内容审核

审核对象包括岗位描述、简历内容、运营内容、AI 输出报告、公开情报线索。审核结果包括通过、驳回、人工复核。驳回必须填写原因。

| 页面交互 | 页面状态、按钮行为、空状态、加载状态、异常状态符合设计 |
| 数据正确 | 用户、企业、岗位、画像、投递、面试、报告、通知可正确保存和读取 |

### 20.1 MVP 验收闭环

求职者闭环：注册/登录 -> AI 顾问访谈 -> 上传简历 -> 生成职业画像 -> 构建人才图谱 -> 获取岗位推荐 -> 查看匹配解释/报告 -> 完成模拟面试 -> 生成成长计划

企业招聘闭环：企业注册/认证 -> 发布岗位/上传 JD -> AI 岗位访谈 -> 生成岗位画像 -> 智能匹配候选人 -> 查看候选人详情 -> 查看推荐报告 -> 预约面试/反馈

平台管理闭环：查看数据概览 -> 管理用户/企业 -> 审核企业认证 -> 审核运营内容/业务内容 -> 查看画像和图谱 -> 监控匹配结果 -> 调整基础配置

AI 能力闭环：输入简历/JD/访谈内容 -> AI 解析 -> 画像生成 -> 图谱构建 -> 匹配推荐 -> 报告生成 -> 结果展示与入库
